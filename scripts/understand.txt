dnsdb in  Architecture

1) Current Architecture (Simplified)

Control Plane:
- UI / API (your control plane / NIOS)
- Stores DNS records in DB
- Handles CRUD, audit logs, imports

Data Plane:
- Authoritative DNS servers
- Answer DNS queries

State Storage:
- Relational DB
- Snapshots
- Change logs

---

2) Where dnsdb Fits

dnsdb replaces the State Storage + Change Management layer inside the Control Plane.

It does NOT replace DNS servers.

---

3) Before vs After

Before:
[ UI / API ]
      ↓
[ Mutable DB ]
      ↓
[ Zone files ]
      ↓
[ DNS servers ]

After:
[ UI / API ]
      ↓
[ dnsdb (Merkle DAG, commits) ]
      ↓
[ Zone snapshot (HEAD) ]
      ↓
[ DNS servers ]

---

4) What Gets Replaced

- Mutable DB tables → replaced by append-only commits
- Snapshots/backups → replaced by commit history
- Audit logs → replaced by cryptographic history
- Import/sync logic → replaced by commit-based diffs

---

5) What Stays

- DNS servers
- DNS protocol
- External APIs

---

6) Flow Change

Before:
Update → DB change → regenerate zone → reload DNS

After:
Commit → new version → update HEAD → reload DNS

---

7) Core Shift

From: Mutable state
To: Immutable versioned log

---

8) Why It Matters

- Easier rollback
- Better debugging
- Cleaner sync (NIOS → your control plane)
- Eliminates drift and partial failures

---

9) One-liner

dnsdb replaces the mutable database with a versioned, immutable source of truth for DNS state.
